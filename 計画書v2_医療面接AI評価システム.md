# 医療面接AI評価システム 計画書 v2

> 最終更新：2026年3月31日
> 対象：朝日医療大学校 柔道整復学科 臨床実習

## 1. プロジェクト概要

### 1-1. 目的

朝日医療大学校 柔道整復学科の臨床実習（整骨院）において、学生が実施した医療面接をAIが自動評価し、教員との振り返りを支援する**セルフチェックシステム**を構築する。

### 1-2. コンセプト

```
学生がPlaud Note Pinで医療面接を録音（整骨院にて）
        ↓
教員のiPhone（Plaudアプリ）でBluetooth同期
→「医療面接匿名化テンプレート」で匿名化済みテキストを自動生成
        ↓
PCのブラウザ(app.plaud.ai)で匿名化済みテキストをコピー→WebAppに貼り付け
        ↓
学生が自己評価を入力（AI評価の前に・音声入力対応）
        ↓
Claude Sonnet 4.6（AWS Bedrock）が評価基準に基づき自動判定
        ↓
大画面モニターで教員と学生が一緒に結果確認・振り返り
        ↓
結果をGmailで学生に送信（後から見返す用）
```

### 1-3. 技術スタック

| 項目 | 採用技術 | 選定理由 |
|------|----------|----------|
| 録音・文字起こし・匿名化 | Plaud Note Pin＋Plaudアプリ（iPhone）＋匿名化テンプレート | 既に所有、月300分無料枠、カスタムテンプレートで匿名化を自動化、PCブラウザ(app.plaud.ai)で閲覧可能 |
| バックエンド／ホスティング | GAS（Google Apps Script） | 無料、Google Workspace連携が容易 |
| AIエンジン | Claude Sonnet 4.6（AWS Bedrock） | 高精度な評価、教育的で温かいフィードバック、JSON出力対応、東京リージョン（低遅延） |
| データベース | Google Sheets | GASとのネイティブ連携、教員が直接確認可能 |
| フロントエンド | GAS WebApp（HTML Service） | 追加サーバー不要、デプロイが簡単 |
| メール送信 | GmailApp.sendEmail() | GAS標準機能、追加設定不要 |
| APIキー管理 | AWS Bedrock長期APIキー | GASのスクリプトプロパティに格納 |

### 1-4. システムパターン

**パターンC（フル自動化）** を採用。すべての出力タイプを実装する。

| 出力タイプ | 内容 |
|-----------|------|
| 自動採点 | Good / Partial / Poor の三段階判定 |
| フィードバック | レベル別トーンでの改善アドバイス |
| 技法紐づけアドバイス | テキスト教材の該当ページ・技法と対応づけ |
| スコアリング | 100点満点＋グレード（Level 2以上） |

## 3. テキスト転送フローと運用フロー

### 3-1. テキスト転送フロー

```
Plaud Note Pin（面接現場で録音）
    ↓  Bluetooth接続
教員のiPhone（Plaudアプリ →「医療面接匿名化テンプレート」で匿名化済みテキスト生成）
    ↓  Wi-Fi経由でクラウド同期（自動）
整骨院のPC（ブラウザ2タブ）
    ├─ タブ1: app.plaud.ai → 匿名化済みテキストをコピー
    └─ タブ2: WebApp → 貼り付け → 自己評価 → AI評価
         ↓  大画面モニターで振り返り
「結果をメールで送る」→ 学生のGmailに送信
```

**※ 匿名化はPlaudのカスタムテンプレートが自動実行。教員の匿名化作業はゼロ。**

### 3-2. 1人あたりの運用フロー（約15〜25分）

| ステップ | 所要時間 | 内容 | 担当 |
|---------|---------|------|------|
| 1. 面接実施 | 5〜10分 | 学生がPlaud Note Pinの録音ボタンを押して面接、終了後停止 | 学生 |
| 2. 文字起こし＋匿名化 | 3〜5分 | 教員がiPhoneのPlaudアプリでBluetooth同期→匿名化テンプレートで自動生成 | 教員 |
| 3. PCで操作 | 1〜2分 | app.plaud.aiで匿名化済みテキストをコピー→WebAppに貼り付け | 教員 |
| 4. 自己評価入力 | 1〜3分 | PCでチェックボックス＋自由記述は音声入力 | 学生 |
| 5. AI評価実行 | 30秒〜1分 | ボタン押下→結果表示 | 教員or学生 |
| 6. 結果確認・振り返り | 3〜5分 | 大画面モニターで教員と学生が一緒に確認 | 教員＋学生 |
| 7. メール送信 | 30秒 | 「結果をメールで送る」ボタンで学生のGmailに送信 | 教員or学生 |

## 5. 評価レベル設計（3段階）

### 5-1. レベル一覧

| レベル | 対象 | 項目数 | スコア形式 | フィードバックのトーン |
|--------|------|--------|-----------|---------------------|
| Level 1（入門） | 初学者 | 10項目 | 点数なし、チェック数のみ（✅ 7/10） | コーチが応援するトーン |
| Level 2（基本） | 中級者 | 30項目 | 100点満点＋グレード（S/A/B/C/成長中） | バランス型 |
| Level 3（応用） | 上級者 | 68項目 | 詳細分析＋問いかけ形式 | 自己省察促進型 |

### 5-2. レベル別フィードバック詳細

#### Level 1（入門）

- **トーン：** 「コーチが応援する」トーン。できたことを先に多く伝える
- **改善点：** 1つだけに絞る
- **台本付き：** そのまま使えるセリフを提示
- **初期表示：** チェック数＋次の1アクション

```
例：
✅ 7/10 できました！ とてもよくできています！

【次にチャレンジ！】
「繰り返し」をやってみましょう

患者：「肩が痛いんです」
    ↓
あなた：「肩が痛いのですね。それはお辛いですね。」

💡 患者さんの言葉をそのまま繰り返すだけでOK！
```

#### Level 2（基本）

- **トーン：** バランス型。良い点3つ＋改善点3つ
- **改善点の構成：** 「なぜ重要か」の理由＋具体例を付与
- **最優先アクション：** 1つを大きく表示。詳細は折りたたみ
- **初期表示：** スコア＋グレード＋カテゴリ別＋最優先アクション1つ

#### Level 3（応用）

- **トーン：** 自己省察促進型。問いかけ形式
- **問いかけ例：** 「この場面で〜できた可能性はありませんか？」
- **答えは折りたたみに格納：** 自分で考えさせる
- **初期表示：** スコア＋対話分析＋自己省察の問い

### 5-3. レベル選択方式（ハイブリッド）

システムが推奨＋学生が自由に変更可能。

| 条件 | 推奨レベル |
|------|-----------|
| 初回 | 必ずLevel 1がデフォルト |
| Level 1で3回以上8/10以上 | Level 2を推奨 |
| Level 2で3回以上80点以上 | Level 3を推奨 |

※学生はいつでも手動でレベルを変更可能

## 6. 評価判定とスコアリング

### 6-1. 三段階判定

| 判定 | 表示 | 意味 |
|------|------|------|
| Good | ✅ | 適切に実施 |
| Partial | 🔶 | 試みはあるが不十分 |
| Poor | ➖ | 実施されていない（「まだやっていない＝これからできる余白」の印象） |

※ Poorの表示に警告マーク（⚠️）は使わない。「➖」（横線）で「これからできる余白」を表現する。

### 6-2. グレード設計（Level 2以上）

| グレード | 点数 | 表示 |
|---------|------|------|
| S | 90〜100点 | 優秀 |
| A | 80〜89点 | 良好 |
| B | 70〜79点 | 合格 |
| C | 60〜69点 | あと一歩 |
| 成長中 | 59点以下 | 成長中 |

※ D廃止。低スコア時は「成長中」表記とし、**できた点を先に3つ以上褒め、改善点は1つだけ**に絞る。

### 6-3. スコアリング配点（100点満点）

```
面接の構造・流れ ─────── 15点（Step1〜4の順序と網羅性）
オープニング ──────────── 10点（挨拶・自己紹介・確認・同意）
コミュニケーション技法 ── 25点
  ├─ 傾聴（促し・繰り返し・要約・確認）  10点
  ├─ 共感・支持                            8点
  └─ 質問技法（OQ/CQ バランス）            7点
医学的情報の聴取 ──────── 30点
  ├─ 主訴・現病歴（必須7項目）           20点
  └─ 既往歴・家族歴・社会歴               10点
解釈モデルの聴取 ──────── 10点（LET'S HEAR）
締めくくり ────────────── 10点（最終要約・聞き漏らし・挨拶）

【追加加点/減点】
＋ 効果的な話法の使用（連呼法・ダブルバインド等）  最大＋5点
－ 専門用語の多用                                   最大-5点
－ 患者の話の遮り                                   最大-5点
－ 不適切な言葉遣い                                 最大-5点
```

### 6-4. 情報の段階的開示

最初に表示するのは最小限。詳細は折りたたみで格納する。

| レベル | 初期表示 | 折りたたみに格納 |
|--------|---------|-----------------|
| Level 1 | チェック数＋次の1アクション | できた項目の一覧 |
| Level 2 | スコア＋グレード＋カテゴリ別＋最優先アクション1つ | 項目別詳細、良かった点、改善詳細、対話分析 |
| Level 3 | スコア＋対話分析＋自己省察の問い | 68項目詳細、テキスト教材対応、ターンごと評価 |

## 7. 自己評価とAI評価の組み合わせ

### 7-1. 概要

AI評価の**前に**学生が自己評価を入力する。結果画面で「自己評価 vs AI評価」を並列表示し、ギャップを分析する。

### 7-2. レベル別の自己評価入力

| レベル | 入力内容 | 所要時間 |
|--------|---------|---------|
| Level 1 | 10項目のチェックボックス | 約1分 |
| Level 2 | カテゴリ別5段階評価＋自由記述 | 約2〜3分 |
| Level 3 | Level 2＋「意識して取り組んだこと」「うまくいかなかった場面」の自由記述 | 約3分 |

### 7-3. ギャップ分析

| ギャップの種類 | 意味 | AIコメント例 |
|--------------|------|-------------|
| 自己高・AI低 | 自己評価が高いがAIは低い | 「この項目は意識していたようですが、テキストからは十分に確認できませんでした。次回は...」 |
| 自己低・AI高 | 自己評価が低いがAIは高い | 「自分では気づいていないかもしれませんが、実はよくできています！」 |
| 一致 | 自己評価とAIが一致 | 「自己認識が正確です」 |

- **一致率**を表示（例：「自己評価とAI評価の一致率：70%」）
- ギャップの**教育的意味**をAIがコメント

## 8. 評価基準（テキスト教材ベース）

### 8-1. 準拠教材

- **教科書：** 「臨床観察実習〜柔整師のための医療面接〜」（森経介著）
- **ページ数：** 85ページ、全6回構成
- **主な内容：** 38種の技法、LET'S HEARフレームワーク、4ステップ（導入→展開→掘り下げ→終了）

### 8-2. 知識ベースの格納方法

- `EvaluationPrompt.gs` に**要約版**を格納
- トークン数の最適化のため全文は入れない
- 各技法の定義、判定基準、具体例を要約して含める

### 8-3. テキストから評価できる項目（主要評価対象）

| カテゴリ | 主な評価項目 |
|---------|------------|
| 面接の流れ | Step1→2→3→4の順序性 |
| オープニング | 挨拶、自己紹介、患者確認（フルネーム）、面接目的説明と同意取得 |
| 質問技法 | 開かれた質問、閉ざされた質問、OQ/CQ比率、焦点を絞った質問 |
| 傾聴技法 | 促し、繰り返し、要約、確認、バックトラッキング |
| 共感・支持 | 共感の言葉がけ、支持・勇気づけ |
| 情報聴取 | 主訴、発生機序、部位・性状・程度・経過、増悪寛解因子、随伴症状、ADL、既往歴・家族歴・社会歴 |
| 解釈モデル | LET'S HEARの要素検出 |
| 言葉遣い | 敬語の適切さ、専門用語の多用チェック |
| 締めくくり | 最終要約、聞き漏らし確認、締めの挨拶 |

### 8-4. AI評価の限界（学生に明示する）

- **非言語コミュニケーション（全体の2/3）は評価対象外**
  - 身だしなみ、視線・表情、姿勢・態度、座席配置など
  - テキスト分析では評価できないことを結果画面に明記する
- この制限を学生に正しく理解させることが重要

| レスポンス時間 | 約15〜20秒 |

### 9-2. エラーハンドリング

- 4〜6人/半日 → レート制限に対して十分余裕あり
- 複雑なキュー/遅延は不要
- **最低限の自動リトライ（エラー時）のみ**実装

```javascript
// 自動リトライの実装イメージ
function callBedrockWithRetry(payload, maxRetries = 3) {
  const BEDROCK_URL = 'https://bedrock-runtime.ap-northeast-1.amazonaws.com/model/global.anthropic.claude-sonnet-4-6/converse';
  const API_KEY = PropertiesService.getScriptProperties().getProperty('BEDROCK_API_KEY');

  for (let i = 0; i < maxRetries; i++) {
    const options = {
      method: 'post',
      contentType: 'application/json',
      headers: { 'Authorization': 'Bearer ' + API_KEY },
      payload: JSON.stringify(payload),
      muteHttpExceptions: true
    };
    const response = UrlFetchApp.fetch(BEDROCK_URL, options);
    if (response.getResponseCode() === 200) {
      return JSON.parse(response.getContentText());
    }
    if (response.getResponseCode() === 429) {
      Utilities.sleep(10000 * (i + 1));
      continue;
    }
    throw new Error('API Error: ' + response.getResponseCode());
  }
  throw new Error('リトライ上限に達しました');
}
```

## 11. プライバシーとデータ保護

### 11-1. 基本方針

| 項目 | 対応 |
|------|------|
| Plaudクラウド | 録音データ・文字起こしはPlaudサーバーに保存（教員アカウントで管理） |
| AWS Bedrock（従量課金） | 有料利用のためデータはモデル改善に使用されない。匿名化済みテキストのみ送信 |
| 匿名化 | **Plaudの「医療面接匿名化テンプレート」で自動実行**（教員の手動作業ゼロ） |
| 同意書 | 作成しない |
| 評価結果の保存 | 匿名化済みテキスト＋スコア＋フィードバックをGoogle Sheetsに保存 |
| WebAppアクセス制限 | **「自分のみ」**に設定（教員以外はアクセス不可） |

### 11-2. Plaud匿名化テンプレート

Plaudのカスタムテンプレート機能で以下の匿名化ルールを自動適用：

- 患者の氏名 → 「患者A」に自動置換
- 学生の氏名 → 「術者」に自動置換
- 住所・電話番号 → 「[削除]」に自動置換
- 整骨院名・施設名 → 「[施設]」に自動置換
- タイムスタンプ削除、「術者：」「患者A：」の対話形式で出力

### 11-3. AI評価の位置づけ

- AIの評価はあくまで**参考・自己学習支援**であり、成績評価には直接使用しない
- 最終的な評価は教員が行う
- 非言語コミュニケーション（全体の2/3）は評価対象外であることを学生に明示する

## 13. 重要な知見と設計上の注意点

### 13-1. AI評価の限界への対処

| 課題 | 対処 |
|------|------|
| 非言語コミュニケーション（全体の2/3）は評価不可 | 結果画面に「テキストベースの評価です。表情・姿勢・声のトーンは含まれません」と明記 |
| 患者の寡黙度による評価ブレ | プロンプトで「患者の発話が少ない場合、閉ざされた質問が増えるのは適切な対応」と指示 |
| 模擬面接と実患者面接の違い | 面接種別をメタ情報として入力し、プロンプトの評価基準を微調整 |

### 13-2. キャリブレーション

- **本格運用前に教員評価済み面接10件でAI精度を検証する**
- 教員の手動評価とAI評価のスコア差を確認
- 大きな乖離がある項目はプロンプトを調整
- キャリブレーション結果を記録し、継続的に精度を改善

### 13-3. 評価基準の外部化（将来）

- 現状：`EvaluationPrompt.gs` にハードコード
- 将来：Google Sheetsに評価基準を移し、教員がコード不要で更新可能にする
- これにより、教材変更や評価項目の追加・修正が容易になる

### 13-4. 低スコア時の心理的配慮

- 「D」「不合格」「再実施」などのネガティブ表現は使わない
- 59点以下は「成長中」と表記
- **できた点を先に3つ以上褒め、改善点は1つだけに絞る**
- Level 1では褒めを多くし、改善点の台本（そのまま使えるセリフ）を付ける

## 付録C：レベル推奨ロジック

```javascript
function getRecommendedLevel(history) {
  // 初回は必ずLevel 1
  if (history.length === 0) return 1;

  // Level 1で3回以上8/10以上 → Level 2推奨
  const level1Results = history.filter(h => h.level === 1);
  const level1HighCount = level1Results.filter(h => h.checkCount >= 8).length;

  if (level1HighCount >= 3) {
    // Level 2で3回以上80点以上 → Level 3推奨
    const level2Results = history.filter(h => h.level === 2);
    const level2HighCount = level2Results.filter(h => h.score >= 80).length;
    if (level2HighCount >= 3) return 3;
    return 2;
  }

  return 1;
}
```
